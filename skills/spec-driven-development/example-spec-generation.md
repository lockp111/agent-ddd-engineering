## Input: Phase 4 Tech Decision (excerpt)

> **Interface Type:** gRPC for all internal context-to-context communication (Order <-> Inventory, Order <-> Payment). REST for external-facing Order API (CreateOrder, CancelOrder, GetOrder exposed to frontend).

## Step 1: INVENTORY

Read Phase 3 contracts + Phase 4 tech decisions. Map each interface to a spec format:

| Contract Interface | Methods | Spec Format | Rationale |
|:---|:---|:---|:---|
| OrderAppServicePort | CreateOrder, CancelOrder, GetOrder | **OpenAPI** (REST) | Phase 4: external-facing API |
| InventoryServicePort | Reserve, Release | **Proto** (gRPC) | Phase 4: internal context-to-context |
| PaymentServicePort | Authorize, Refund | **Proto** (gRPC) | Phase 4: internal context-to-context |

**Key decision:** Phase 4 explicitly maps internal to gRPC, external to REST. No guessing needed.

## Step 2: SCAFFOLD

Create file structure following the one-aggregate-per-set rule:

```
specs/
└── order/
    ├── order/
    │   ├── order-service.yaml          # OpenAPI — external API (≤200 lines)
    │   └── order-types.yaml            # OpenAPI — shared request/response types
    ├── inventory/
    │   ├── inventory-service.proto      # gRPC — Reserve, Release
    │   └── inventory-types.proto        # gRPC — request/response + error types
    ├── payment/
    │   ├── payment-service.proto        # gRPC — Authorize, Refund
    │   └── payment-types.proto         # gRPC — request/response + error types
    └── shared/
        └── common-errors.proto          # Infrastructure error patterns
```

Each file targets ≤200 lines. The Order aggregate's external API uses OpenAPI (YAML); the Inventory and Payment ports use Proto.

### inventory-types.proto

```protobuf
// Source: phase-3-contracts.md#InventoryServicePort (types)
// Generated by SDD Generate Mode | 2026-03-17
syntax = "proto3";

package order.inventory.v1;

import "google/protobuf/timestamp.proto";

// --- Reserve ---

message ReserveInventoryRequest {
  string order_id = 1;                               // required
  repeated ReserveInventoryItem items = 2;            // required, ≥1
}

message ReserveInventoryItem {
  string sku = 1;                                     // required, non-empty
  int32 quantity = 2;                                 // required, ≥1
}

message ReserveInventoryResponse {
  string reservation_id = 1;
  string order_id = 2;
  ReservationStatus status = 3;
  repeated ShortageItem shortage_items = 4;           // populated only if SHORTAGE
}

enum ReservationStatus {
  RESERVATION_STATUS_UNSPECIFIED = 0;
  RESERVED = 1;
  SHORTAGE = 2;
}

message ShortageItem {
  string sku = 1;
  int32 available_quantity = 2;
  int32 requested_quantity = 3;
}

// --- Release ---

message ReleaseInventoryRequest {
  string reservation_id = 1;                          // required
}

message ReleaseInventoryResponse {
  bool released = 1;
}

// --- Domain Errors ---

// Domain errors for InventoryService
// Used by: Reserve (INSUFFICIENT_STOCK, INVALID_SKU, RESERVATION_CONFLICT)
// Used by: Release (RESERVATION_NOT_FOUND, ALREADY_RELEASED)
enum InventoryError {
  INVENTORY_ERROR_UNSPECIFIED = 0;
  INSUFFICIENT_STOCK = 1;
  INVALID_SKU = 2;
  RESERVATION_CONFLICT = 3;
  RESERVATION_NOT_FOUND = 4;
  ALREADY_RELEASED = 5;
}
```

**Key decisions during FILL:**

1. **Error types generated:** `INSUFFICIENT_STOCK` and `INVALID_SKU` for Reserve (derived from contract's `ShortageItem` and SKU field). `RESERVATION_CONFLICT` added because concurrent reservations are a known domain error. `RESERVATION_NOT_FOUND` and `ALREADY_RELEASED` for Release (idempotency guards).
2. **Field constraints:** `quantity` is `int32` with business rule >=1 (documented in comment). `sku` is non-empty string.
3. **Traceability:** Every service and RPC has a `// Source:` comment linking back to the contract.
