# Merge Mode Walkthrough: Reconciling Human Edits with Updated Contracts

> This document walks through SDD Merge Mode when existing Proto files have been manually edited by a human, and updated Phase 3 contracts introduce a new method. Demonstrates THREE-WAY DIFF, CONFLICT resolution, EXTRA item handling, and manifest update.

---

## Context

The team previously ran SDD Generate Mode for the Order context's Inventory port. Since then:

1. **Human edit:** A developer added a `CheckStock` convenience RPC and a business comment explaining reservation TTL behavior.
2. **Contract update:** Phase 3 contracts were revised to add a new `ExtendReservation` method, and the `Release` method's return type changed from `bool` to a structured response.

The `specs/` directory already has files, so SDD enters **Merge Mode** automatically.

---

## Step 1: THREE-WAY DIFF

For each spec section, compare three versions:

| Section | Common Ancestor (manifest hash) | Current File (disk) | Contract-Derived (new) | Delta |
|:---|:---|:---|:---|:---|
| `rpc Reserve(...)` | Original | Unchanged | Unchanged | No change |
| `rpc Release(...)` | `returns (ReleaseInventoryResponse)` | Unchanged | `returns (ReleaseResult)` — return type changed | Contract changed |
| `rpc CheckStock(...)` | Not present | **Human added** | Not in contracts | EXTRA item |
| `rpc ExtendReservation(...)` | Not present | Not present | **New in contracts** | Contract added |
| `ReleaseInventoryResponse` | `bool released = 1;` | Unchanged | Replaced by `ReleaseResult` with 3 fields | Contract changed |
| `CheckStock{Request,Response}` types | Not present | **Human added** | Not in contracts | EXTRA items |

### Diff Summary

```
UNCHANGED:  Reserve RPC + types
CONTRACT:   Release return type changed (ReleaseInventoryResponse → ReleaseResult)
CONTRACT:   New RPC ExtendReservation + types
EXTRA:      CheckStock RPC + types (human added, no contract source)
```

---

## Step 2: RESOLVE

Apply merge rules from the skill:

### Case 1: Reserve — No change / No change → **SKIP**

Both sides unchanged. No action needed.

### Case 2: Release return type — No human edit / Contract changed → **AUTO-UPDATE**

Human did not touch Release. Contract changed the return type. Auto-update:

```protobuf
  // Source: phase-3-contracts.md#InventoryServicePort.Release
  // Domain errors: RESERVATION_NOT_FOUND, ALREADY_RELEASED
  rpc Release(ReleaseInventoryRequest) returns (ReleaseResult);
```

New type added to `inventory-types.proto`:

```protobuf
message ReleaseResult {
  string reservation_id = 1;
  google.protobuf.Timestamp released_at = 2;
  int32 items_released = 3;
}
```

Old `ReleaseInventoryResponse` marked as reserved (not deleted — breaking change protection):

```protobuf
// DEPRECATED: replaced by ReleaseResult per contract update 2026-03-17
// reserved "ReleaseInventoryResponse";
```

### Case 3: ExtendReservation — New in contracts → **AUTO-ADD**

New contract method with no existing spec entry. Auto-add:

```protobuf
  // Source: phase-3-contracts.md#InventoryServicePort.ExtendReservation
  // Domain errors: RESERVATION_NOT_FOUND, RESERVATION_EXPIRED, EXTENSION_LIMIT_EXCEEDED
  rpc ExtendReservation(ExtendReservationRequest) returns (ExtendReservationResponse);
```

New types:

```protobuf
message ExtendReservationRequest {
  string reservation_id = 1;               // required
  int32 extend_minutes = 2;                // required, ≥1, ≤60
}

message ExtendReservationResponse {
  string reservation_id = 1;
  google.protobuf.Timestamp new_expiry = 2;
}
```

New error types added to the `InventoryError` enum:

```protobuf
  RESERVATION_EXPIRED = 6;
  EXTENSION_LIMIT_EXCEEDED = 7;
```

### Case 4: CheckStock — Human added / Not in contracts → **EXTRA — WARNING**

```
WARNING: rpc CheckStock and associated types (CheckStockRequest, CheckStockResponse,
StockLevel) exist in spec but have no contract source.

Is this an intentional addition? If yes, consider adding CheckStock to Phase 3
contracts to maintain traceability. If no, these items may need removal.

Action: Keeping human version. Flagged in spec-manifest.md Untraced Items section.
```

The human's CheckStock RPC, its types, and the business comment about reservation TTL are **preserved exactly as written**.

## CONFLICT Example: What If Human Had Also Changed Release?

Suppose the human had modified the Release return type to add a `reason` field:

```protobuf
// Human version (hypothetical)
message ReleaseInventoryResponse {
  bool released = 1;
  string release_reason = 2;    // [HUMAN ADDED] audit trail
}
```

And the contract also changed Release to return `ReleaseResult`:

| Section | Human edited? | Contract changed? | Action |
|:---:|:---:|:---:|:---|
| Release return type | **Yes** (added field) | **Yes** (new type) | **STOP** — conflict |

```
CONFLICT: Both human and contract modified the Release return type.

  Human version:  ReleaseInventoryResponse { bool released; string release_reason; }
  Contract version: ReleaseResult { string reservation_id; Timestamp released_at; int32 items_released; }

These are incompatible changes. Only the human can decide:
  (a) Keep human version and update contracts to match
  (b) Accept contract version (human's release_reason field will be lost)
  (c) Merge both — add release_reason to the new ReleaseResult type

Waiting for human decision before proceeding.
```

**The agent STOPS here. It does not pick a side.**
