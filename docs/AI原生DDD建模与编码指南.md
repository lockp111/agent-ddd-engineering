# AI 原生时代的 DDD 建模与编码实操指南 (AI-Native DDD Process)

> **导读**：
> 传统的 DDD（领域驱动设计）旨在解决人类团队的认知对齐、沟通成本和组织架构问题。
> 但在 AI 编程时代（LLM Agent / Cursor / Devin），DDD 的架构与组织原则正在被重新应用——它们成为了强大的 **Prompt 工程框架、上下文管理器和大模型幻觉抑制器**。
>
> 本指南将 DDD 的架构和组织原则应用于 AI 编程场景，指导你如何利用大模型处理复杂业务系统。
>
> ⚠️ **重要声明**：本指南侧重 AI 编程的工程实践，不能替代传统 DDD 中与领域专家持续协作探索的核心实践。AI 辅助的"事件风暴"和"上下文划分"是对人类协作建模的**补充**，而非替代。

---

## 1. 范式转换：人类 DDD vs AI 原生 DDD

在让 LLM 编写复杂业务系统时，最大的灾难往往是**上下文污染（Context Contamination）**和**超出窗口限制（Context Window Limit）**。将 DDD 应用于 AI 编程，核心在于概念的平移：

| 传统 DDD 概念 | AI 编程场景下的实操映射 | 解决的 AI 编程痛点 | ⚠️ 原始概念未覆盖的语义 |
| :--- | :--- | :--- | :--- |
| **工作坊/便签 (EventStorming)** | **结构化知识抽取 Prompt** | 解决大模型面对模糊需求时的无从下手。 | 缺少多角色跨职能的协作碰撞与 hot spots 发现。 |
| **限界上下文 (Bounded Context)** | **Agent 的物理隔离舱 / System Prompt** | 防止 LLM 在写 A 模块代码时混入 B 模块的逻辑（幻觉）。 | 未涵盖语义边界定义及 BC 间关系模式（ACL、Customer-Supplier 等）。 |
| **统一语言 (Ubiquitous Language)** | **强制术语约束字典** | 防止 LLM 在多轮对话中对同一个实体产生"词汇漂移"（如 User/Account 混用）。 | 未涵盖与领域专家协商演进语言的社会过程。 |
| **跨团队契约 (Context Mapping)** | **API/Event 接口契约生成** | 确保多个独立代码生成任务的最终拼装不出错。 | 未涵盖战略关系模式（ACL、Conformist、Shared Kernel、Open Host Service 等）。 |
| **六边形/洋葱架构** | **分步生成的隔离沙盒** | 避免 LLM 一次性生成包含业务、SQL 和 HTTP 的面条代码导致崩溃。 | 未显式表达依赖反转（dependencies point inward）的核心架构原则。 |

## 2. AI 驱动的 DDD 五步法流

### Phase 1: 知识注入与领域抽取 (Knowledge Extraction)

**目标**：将非结构化的人类业务需求，转化为 LLM 能够精确理解的结构化领域知识。

*   **人类动作**：提供业务 PRD、需求文档或业务痛点的口述文字。
*   **AI 动作**：让 LLM 扮演领域专家，执行"文本版事件风暴"。
*   **Prompt 策略**：
    > "你是一个资深领域专家。请阅读以下业务需求，提取出系统中的**领域事件（Domain Events）**、**触发事件的命令（Commands）**以及**执行命令的角色（Actors）**。请以 Markdown 表格的形式输出结构化清单，不要遗漏异常分支。"
*   **输出物**：结构化的《领域事件清单》。在此阶段，LLM 帮助人类理清了需求盲区。

> **⚠️ 注意**：LLM 生成的事件清单应由**领域专家审核确认**。AI 无法替代多角色协作发现中产生的洞察碰撞，建议将 AI 产出作为领域专家工作坊的讨论起点，而非最终结论。

### Phase 2: 上下文切分与防污染 (Context Partitioning)

**目标**：为 LLM 划定认知边界，确保后续生成代码时，LLM 只关注当前上下文，避免幻觉。

*   **人类动作**：审查 AI 提取的事件清单。
*   **AI 动作**：基于高内聚低耦合原则，对事件和实体进行聚类，划分限界上下文（Bounded Context）。
*   **Prompt 策略**：
    > "基于上述事件清单，请按照 DDD 原则划分『限界上下文』。对于每个上下文，说明其核心职责（Inputs/Outputs）。请注意：订单履约与支付结算必须是不同的上下文。"
*   **工程应用**：**从这一步开始，针对每个限界上下文，应该开启独立的 AI 会话（或分配独立的 Agent）**。绝不能在同一个窗口让 AI 同时写两个限界上下文的代码。

> **⚠️ 注意**：上下文边界的划分应由**领域专家和架构师共同审定**。LLM 倾向于按技术层而非业务语义切分边界，需要人类判断确保边界反映真实的业务领域结构。

### Phase 3: 将统一语言转化为强约束 Prompt (Language as Prompt)

**目标**：消除 LLM 编码过程中的"词汇漂移"。

*   **AI 动作**：为当前独立上下文生成《中英文术语对照表》。
*   **Prompt 策略**：
    > "为『仓储上下文』生成统一语言词汇表。在接下来的所有对话和代码生成中，涉及货物的接收者，必须且只能使用 `Receiver`，绝对禁止使用 `User`、`Customer` 或 `Buyer`。若违反此规则，视为严重故障。"
*   **工程应用**：将这份词汇表作为该限界上下文**全局 System Prompt** 的一部分（在 Cursor 等工具中可将其放入 `.cursorrules` 文件中）。

### Phase 4: 契约优先的交互设计 (Contract-First Connection)

**目标**：在编写具体逻辑前，先让 LLM 生成不同上下文之间的交互标准，屏蔽实现细节。

*   **AI 动作**：基于领域事件，生成 OpenAPI 规范（同步调用）或 AsyncAPI 契约（异步事件）。
*   **Prompt 策略**：
    > "仓储上下文需要监听订单上下文的『OrderPaid』事件。请不要写任何业务逻辑代码，只使用 TypeScript 定义该事件的 Interface 契约（Payload 结构），以及暴露给外部调用的 API 签名。"
*   **工程应用**：有了契约，你就可以并行让不同的 AI 会话分别去实现契约的提供方和消费方。

> **补充：上下文映射关系模式**
> 在生成契约之前，建议先确定各限界上下文之间的**上下文映射关系模式**，因为不同关系模式下的契约设计策略截然不同：
> - **Customer-Supplier（客户-供应商）**：下游方可以向上游方提出需求，契约需协商确定。
> - **Conformist（遵从者）**：下游方完全采用上游方的模型，不做翻译。
> - **Anti-Corruption Layer（防腐层）**：下游方构建翻译层，隔离上游模型的影响。
> - **Open Host Service / Published Language（开放主机服务/公开语言）**：上游方提供标准化协议供多方消费。
> - **Separate Ways（各行其道）**：两个上下文决定不集成，各自独立实现。
>
> 这些模式直接影响 AI 生成的接口契约的结构和复杂度。参考：[Context Mapping](https://github.com/ddd-crew/context-mapping)

### Phase 5: 隔离式渐进编码 (Isolated Progressive Coding)

**目标**：利用六边形架构，分层向 LLM 索要代码，极大提高生成成功率。

**如果让 LLM 一次性生成全栈代码，往往会得到耦合数据库逻辑的垃圾代码。必须分层生成：**

1.  **纯领域核编码 (The Domain Core)**
    *   **Prompt**："现在开始编写核心领域层。请使用纯 TypeScript 生成聚合根（AggregateRoot）、实体（Entity）和值对象（ValueObject）。**禁止引入任何外部依赖（如数据库、ORM、HTTP 框架）**。所有业务规则必须在聚合根内实现。"
    *   *原因：LLM 最擅长生成无副作用的纯逻辑代码，这种代码的单测覆盖率极高。*
    *   **聚合设计要点**（需在 Prompt 中明确约束 LLM）：
        * **一致性边界**：聚合内的所有对象必须在一个事务中保持一致。
        * **尽量小**：过大的聚合会导致并发竞争；过小会导致一致性缺口。可参考 [Aggregate Design Canvas](https://github.com/ddd-crew/aggregate-design-canvas)。
        * **通过标识引用其他聚合**：聚合之间只持有 ID 引用，不持有直接对象引用。
        * **不变量强制执行**：所有业务规则（不变量）必须在聚合根方法中校验，不允许外部绕过。
2.  **用例层编码 (Application Services)**
    *   **Prompt**："基于刚才生成的聚合根，请编写应用服务层（Use Cases）。使用依赖注入引入 Repository 接口。处理事务边界，并发布领域事件。"
3.  **适配器补全 (Adapters: DB & Web)**
    *   **Prompt**："现在，请基于 Prisma/TypeORM 实现刚才定义的 Repository 接口，并基于 Express/NestJS 生成对应的 RESTful API 控制器。"
4.  **自动测试生成 (Automated Test Generation)**
    *   让 LLM 根据 Phase 1 的事件风暴列表，逐一编写验收测试用例。

## 3. 为什么 AI 编程极其需要 DDD？（深度启示）

### 1. 降低 Cognitive Load（认知负荷） -> 降低 Context Window（上下文窗口）占用
人类处理复杂系统的瓶颈在大脑内存；LLM 的瓶颈在 Context Window 及其注意力机制（Attention Mechanism）的稀释。DDD 的限界上下文完美契合了 LLM 的限制，将大工程拆解为一个个只需极少上下文就能完美生成的"微型项目"。

### 2. 代码可维护性 -> Prompt 的可维护性
如果系统没有清晰的聚合和边界，当业务变化时，你需要给 LLM 极长的 Prompt 来解释前因后果。而在良好的 DDD 架构下，你只需要告诉 LLM："修改 `Order` 聚合根中的计价策略，其余基础设施不变。"

### 3. 六边形架构是 AI 的最爱
在缺乏精确文档上下文时，外部依赖（数据库、第三方 API）容易导致 LLM 产生幻觉。通过强制应用六边形架构，你迫使 LLM 先写纯粹的业务逻辑模型。这种纯净的领域代码极大地提高了 AI 编码的准确性和可测试性。

## 4. 实战建议 (针对 Cursor / Copilot 用户)

1. **为每个 Bounded Context 建立独立的目录**。
2. **在每个目录根部放置该上下文专用的 `.cursorrules`（或 README.md）**。
    * 文件内包含：当前上下文的**统一语言字典**、**允许与禁止的外部依赖**。
3. **先让 AI 帮你写 Interface**。人类审核 Interface 无误后，再让 AI 填充具体实现。
4. **禁止 AI 绕过聚合根直接修改数据**。在 System Prompt 中加入硬性规定："所有状态的修改必须通过聚合根的方法暴露，禁止生成任何贫血模型（Anemic Domain Model）风格的 setter。"
